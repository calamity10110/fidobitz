## 2024-05-18 - Optimized A* path planning implementation
**Learning:** Priority queue-based search algorithms in Python (like A*) degrade rapidly in performance and memory when entire paths are copied and stored inside the queue elements. The `path + [(nx, ny)]` list concatenation on every iteration was causing O(N) operations per step, leading to excessive garbage collection and slow traversal on larger grid maps (e.g. 200x200).
**Action:** Always maintain path history externally using a `came_from` mapping (dictionary linking node to parent) when using `heapq`. Construct the final path only once upon reaching the goal. This provides O(1) queue insertions and drastically reduces memory allocations. I reduced the memory footprint by ~43% (7.7MB to 4.3MB) and cut execution time in half for a 200x200 search.

## 2024-05-20 - Mapping ingestion and parser optimizations
**Learning:** In loops generating millions of iterations or operating in real-time constraint environments, mathematical operations and generator expressions impose considerable overhead. Specifically, Python's `math.sin` and `math.cos` are slow compared to a dictionary lookup for discrete integer domains (-360 to 360). Also, parsing dictionary string keys via `split()` generates intermediate lists which accumulate GC pressure, and `sum(1 for ...)` introduces unneeded generator instantiation overhead. Tests may accidentally validate double-execution bugs if duplicated code goes unnoticed, leading to false-positive assertions.
**Action:** Cache discrete trigonometric states in a static class `dict` map when floating-point precision is unneeded. Refactor hot loops using generator comprehensions (like `sum(...)`) into explicit `for`-loops. Swap string list creations like `.split(",")` with index lookups like `.find(",")` coupled with slicing `[:idx]` to directly cast values. Always verify the source of test failures before modifying them to ensure you aren't masking double-execution bugs.
